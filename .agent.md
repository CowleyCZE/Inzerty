## 🚀 Agent Instructions

### When Creating Components

1. **Always define `Props` interface**
   ```typescript
   interface MyComponentProps {
     title: string;
     onAction?: () => void;
   }
   ```

2. **Add JSDoc comments**
   ```typescript
   /**
    * MyComponent - Does something useful
    * @component
    * @example
    * <MyComponent title="Test" />
    */
   ```

3. **Use React.FC typing**
   ```typescript
   const MyComponent: React.FC<MyComponentProps> = ({ title }) => { ... }
   ```

4. **Export as default**
   ```typescript
   export default MyComponent;
   ```

5. **Create test file**
   ```typescript
   // MyComponent.test.tsx
   describe('MyComponent', () => {
     it('should render', () => { ... })
   })
   ```

### When Creating Hooks

1. **Define return type interface**
   ```typescript
   interface UseMyHookReturn {
     data: string;
     loading: boolean;
     refetch: () => Promise<void>;
   }
   ```

2. **Include loading & error states**
3. **Add cleanup in useEffect**
4. **Use useCallback for handlers**
5. **Document parameters with JSDoc**

### When Creating Services

1. **Use singleton pattern**
   ```typescript
   class MyService { ... }
   export default new MyService();
   ```

2. **No state management** - Just API calls
3. **Proper error handling** - try/catch + logging
4. **Typed parameters and returns**

### When Modifying Backend

1. **Update routes in /backend/src/routes/**
2. **Implement controller in /backend/src/controllers/**
3. **Add service logic in /backend/src/services/**
4. **Update types in types.ts if needed**
5. **Return standard JSON response**
   ```json
   { "success": true, "data": {...} }
   { "success": false, "message": "error" }
   ```

### When Fixing Bugs

1. **Find the root cause** - Don't just patch
2. **Add regression test** - Prevent future bugs
3. **Handle edge cases** - null/undefined checks
4. **Log context** - Include helpful debugging info
5. **Test in multiple browsers** - Cross-browser compat

## 📋 Decision-Making Framework

### Code Quality Decisions

| Situation | Decision |
|-----------|----------|
| Should extract logic to hook? | Yes, if used in 2+ components or complex |
| Should create service? | Yes, for API calls or external integrations |
| Should use Context API? | Yes, for truly global state (config, auth) |
| Should use useState? | Yes, for local component state only |
| Should add test? | Yes, for any public function |
| Should optimize with memo? | Yes, if component has expensive renders |

### Architecture Decisions

| Question | Answer |
|----------|--------|
| Where to add new component? | `/components` or subfolder if grouped |
| Where to add API call? | `/services` as singleton |
| Where to add shared logic? | `/hooks` if complex, `/utils` if simple |
| Where to store UI state? | Custom hook in component |
| Where to store app state? | `useConfig` hook (app-level) |
| How to handle errors? | Try-catch + user feedback |
| How to handle loading? | Loading state in hook return |

## ✅ Agent Quality Standards

### Before Completing Any Task

- [ ] Code compiles without errors (`npm run build`)
- [ ] TypeScript types are complete (no `any`)
- [ ] Tests pass and cover main paths
- [ ] JSDoc comments on public functions
- [ ] Error handling is comprehensive
- [ ] No console.log left in code
- [ ] Commit message follows format
- [ ] Related files are updated

### Code Review Questions

1. **Is it typed?** - All functions have proper types?
2. **Is it tested?** - Unit tests cover main paths?
3. **Is it documented?** - Comments explain why, not what?
4. **Is it performant?** - No unnecessary renders or API calls?
5. **Is it accessible?** - Can keyboard users use it?
6. **Is it maintainable?** - Easy for others to understand?
7. **Is it scalable?** - Will it break with more data?
